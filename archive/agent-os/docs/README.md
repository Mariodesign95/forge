## Reading Order

For a new contributor, read in this order:

1. `mission-protocol.md` — understand the vocabulary
2. `architecture.md` — understand the runtime
3. `brain.md` — understand the memory
4. `security.md` — understand the trust model
5. `budget-policy.md` — understand the cost model

For a security review, start with `security.md`, then `architecture.md` (sandboxing section), then `mission-protocol.md` (governance).

For a product review, start with `mission-protocol.md` (the user-facing contract), then `budget-policy.md` (the user-facing cost surface), then skim the rest.

## Phase 1 Scope Summary

The five documents collectively define:

- **One mission lifecycle** with eight states and explicit transitions
- **One orchestrator** that owns mission state, dispatches tasks, and never executes tools directly
- **One message bus** that all components communicate through
- **Nine agent roles** (Phase 1 minimum: Architect, Planner, Researcher, Coder, QA, Reviewer; Phase 2+: Designer, Marketing, Automation, Operations)
- **One tool registry** with permission levels 1-4
- **One transaction log** that is append-only, tamper-evident, and the source of truth
- **One Brain** with four stores (Project, Skill, User, Mission) and a rigorous skill approval workflow
- **One security model** with four permission levels, sandboxing in two phases, and an explicit threat model
- **One budget policy** with estimation, real-time metering, three thresholds, and automatic fallback routing
- **One autonomous run mode** with mission-scoped consent manifests

## What This Blueprint Does NOT Define (Out of Scope for Phase 1)

- UI design details (component library, motion, accessibility) — to be defined in a separate `ui-design.md` in Phase 1.5
- Specific implementation language choices for the Orchestrator (Node.js vs Rust) — final decision in implementation kickoff
- Marketing site, landing copy, SEO for Forge itself — to be generated by Forge as a self-referential demo mission
- Team collaboration features — Phase 3
- Cloud sync — Phase 4 (optional, never default)
- Marketplace — explicitly cut from the roadmap

## Approval

These five documents are the Phase 1 blueprint. Before any implementation begins:

- [ ] All five documents reviewed by the project owner
- [ ] All five documents reviewed by a second contributor (security review on `security.md` mandatory)
- [ ] Discrepancies with the Master Spec resolved
- [ ] Open questions in each document either answered or explicitly deferred with a target phase

Once approved, these documents become the source of truth. Changes require a new versioned document and a logged decision.
